2. How MIPROv2 optimizes this agent
-----------------------------------
ReAct has two LLM stages DSPy can rewrite:

  1) react            — thought / tool / args loop
  2) extract.predict  — final reasoning + titles from the trajectory

Pipeline in src/react_hover/opt_runner.py:
  baseline eval on devset
  → MIPROv2.compile (bare dspy.ReAct, not SafeAgent)
  → save unique program JSON under artifacts/programs/
  → post-opt eval on same devset

Metric (src/react_hover/metric.py top5_recall):
  - eval time: continuous recall in [0, 1], aggregate as percentage
  - during optimization (trace is set): boolean, perfect recall only
    (recall >= 1.0), so only fully correct title sets are bootstrapped

With max_labeled_demos=0, only bootstrapped (model-generated) demos
are used — no gold labeled I/O demos.

--------------------------------------
Source of truth: artifacts/programs/20260723T220058Z_mipro-light_gpt-5.6-luna.json

  - Instructions: UNCHANGED
    "Find all Wikipedia titles relevant to verifying (or refuting) the claim."
    (+ standard agent/tool boilerplate)

  - Demos: 1 bootstrapped few-shot demo added (augmented: true)

  Live prompt shape:
    baseline: system → user
    after:    system → user(demo) → assistant(demo) → user(real claim)

  Demo claim (train[0]):
    "This director is known for his work on Miss Potter. The Academy of
     Motion Picture Arts and Sciences presents the award in which he was
     nominated for his work in \"Babe\"."

  Gold titles for that claim:
    Miss Potter | Academy Award for Best Director | Chris Noonan

  Demo trajectory (pathological):
    1. search_wikipedia("Miss Potter director Babe Academy Award nomination")
       → Search error: Expecting value: line 1 column 1 (char 0)
    2. search_wikipedia("Chris Noonan Babe Academy Award nomination")
       → Search error: Expecting value: line 1 column 1 (char 0)
    3. finish, asserting Chris Noonan / Best Director without successful lookups

  This teaches: tools failed → invent answer → finish.

3b. extract.predict stage
  - Demos: none (demos: [])
  - Instructions: REWRITTEN by MIPRO (main prompt change)

  Baseline extract objective (one line):
    Find all Wikipedia titles relevant to verifying (or refuting) the claim.

  Optimized extract objective (summary):
    Given claim + completed trajectory, produce evidence-retrieval answer.
    Identify people/works/awards/orgs/events including inferred entities.
    Use trajectory when available; correct unsupported tentative reasoning.
    Return reasoning (identity/relationship chain) and titles (deduplicated
    canonical Wikipedia titles). Prefer disambiguated titles e.g. Babe (film).
    Do not overclaim verification when trajectory only has failed searches.

5. Is there a bug?
------------------
5a. What does NOT look like a bug
  - Optimized program was applied (prompts differ; extract instructions longer)
  - Same dev set for baseline vs after (linked parent_id)
  - Score math is correct (mean recall × 100)
  - program_path in after config matches the saved JSON
  - Flat Δ=0 is a real “no gain,” not a failed load

5b. What DOES look broken or harmful

  (1) Wikipedia tools failing heavily (strongest red flag)
      Traces full of:
        Search error: Expecting value: line 1 column 1 (char 0)
        page fetch failed: Expecting value: line 1 column 1 (char 0)
      Cause: wikipedia client getting empty/non-JSON responses
      (rate limit, block, or flaky network). Intermittent — a later
      manual probe can succeed.

      tools.py turns exceptions into soft observation strings; the agent
      continues and the run still “succeeds.” So:
        - scores can look OK while retrieval is mostly dead
        - MIPRO bootstraps on garbage trajectories
      num_threads=4 during compile increases rate-limit risk.

  (2) Metric accepts tool-free / failed-tool “perfect” traces
      top5_recall only checks gold titles ⊆ pred.titles[:5].
      It does not require successful search/lookup.
      Extract can guess gold titles from model knowledge → bootstrap win
      even when every tool call failed. That is how the Miss Potter demo
      got into the program.

  (3) Bad demo teaches the wrong policy
      Few-shot example on every react call shows failed searches then finish.
      After-run example [4] finishes immediately without tools and still
      scores 1.0 — multi-hop retrieval in name only.

  (4) Exact title match + corpus mismatch
      HoVer gold titles come from wiki17-style abstracts (tutorial ColBERT
      index). This run used live Wikipedia API. Near-miss losses:
        Baboo (airline) vs Baboo
        Eleutherococcus often missing
        List of To Kill a Mockingbird characters vs To Kill a Mockingbird
      Documented in README; still hurts interpretability of “66%.”

  (5) Optimization budget too small for signal (not a code bug)
      train_size=5, dev_size=5, auto=light, max_labeled_demos=0,
      perfect-only metric during opt. Flat scores are easy even when
      prompts change.


6. Bottom-line table
--------------------
Hypothesis                                         Verdict
-------------------------------------------------  -------------------------
Optimized program not applied                      Unlikely
After score miscomputed / wrong baseline           Unlikely
Tools/infra poisoning opt + eval                   Yes — strong evidence
Metric accepts failed-tool perfect traces          Yes — by design, harmful
Bad demo teaches wrong policy                      Yes — consequence of above
DSPy “fake” / no real optimization                 No


7. Why 66.67 → 66.67 is expected here
-------------------------------------
- Retrieval was flaky during both baseline and after.
- Model often compensates with parametric knowledge (especially example 4).
- Only one weak bootstrapped demo + extract instruction rewrite.
- 5-example dev set; 4/5 title lists identical before/after.
- Nothing suggests the optimize path failed to save or reload the program.

9. Possible follow-ups (not done)
---------------------------------
- Harden Wikipedia backend: retries, backoff, hard-fail when search is down
- Per-run tool success rate diagnostic; flag demos with failed observations
- Re-run optimize only after tools look healthy
- Larger train/dev, auto=medium, consider max_labeled_demos > 0
- Prefer ColBERT wiki17 backend when reachable (matches HoVer gold corpus)
